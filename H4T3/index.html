▲ 맨위 Day1 Day2 Day3 총정리

정보처리기사 필기 · 3일 컷 기출 완전정복

2021~2025년 15개 기출(1~3회) 빈출 개념을 분석하여 3일 안에 합격 가능하도록 구성한 흑백 공부용 자료입니다. 5과목 × 20문항 = 총 100문항, 과목당 40점 이상 + 평균 60점 이상이 합격선입니다. 기출 반복 키워드에 HOT 표시.

3일 운영전략
Day 1 — 소프트웨어 설계 + 소프트웨어 개발 (과목1+2) · 집중 4~5h
Day 2 — 데이터베이스 구축 + 프로그래밍 언어 활용 (과목3+4) · 집중 5~6h
Day 3 — 정보시스템 구축관리 (과목5) + 총정리 암기 + 2회분 풀기 · 4h

목차

  1. Day 1 — 과목1 소프트웨어 설계 / 과목2 소프트웨어 개발
  2. Day 2 — 과목3 데이터베이스 구축 / 과목4 프로그래밍 언어 활용
  3. Day 3 — 과목5 정보시스템 구축관리 + 총정리
  4. 최종 요약·암기카드 (시험 직전용)
  5. 자주 틀리는 함정 포인트
합격 공부법 (교육방법론 적용)
  • 간격 반복(Spaced Repetition) — 각 섹션의 "체크포인트"를 Day가 끝나고 한 번, 다음 날 아침 한 번, 시험 직전 한 번 복습.
  • 능동회상(Active Recall) — 개념을 읽고 바로 책 덮고 키워드만 떠올려본 뒤 다시 확인.
  • Chunking — 영문 약어는 알파벳별 풀네임으로 3~5개씩 묶어서 암기.
  • Feynman — 한 줄로 스스로 설명 가능한지 점검. 설명 못 하면 다시 읽기.
  • 기출 우선 — 개념보다 기출문제 유형을 먼저 노출하면 개념이 빠르게 고정됨.
DAY 1

과목 1 · 소프트웨어 설계 & 과목 2 · 소프트웨어 개발

총 분량 4~5시간 · 핵심 키워드 80개

제 1 과목 · 소프트웨어 설계

1-1. 소프트웨어 생명주기 모형 HOT

폭포수 모델 — 전 단계 완료 후 다음 단계. 되돌리기 어려움.
나선형 — 4분면 반복. 위험 관리가 핵심. 대규모 프로젝트에 적합.
  • 폭포수(Waterfall) — 고전적, 단계적 진행, 변경 어려움. "타당성→요구→설계→구현→테스트→유지보수".
  • 프로토타입(Prototyping) — 시제품으로 요구사항 명확화. 최종 결과물 일부를 먼저 의뢰자에게 보여주는 모델. 요구사항 반영 가능(기출 함정: "반영 불가능" = 틀림).
  • 나선형(Spiral) — Boehm 제안. 위험분석이 핵심. 계획→위험분석→개발→평가를 반복.
  • V-모델 — 폭포수 확장, 각 단계별 검증·확인(테스트) 대응.
  • 애자일(Agile) — 변화 대응, 반복적·점진적.
    애자일 4대 가치 : 개인&상호작용 > 프로세스/도구, 작동 SW > 문서, 고객협업 > 계약협상, 변화대응 > 계획준수. (기출: "문서 > 작동 SW" 는 틀림)
생명주기 = "폭프나 V 애" → 폭포수·프로토타입·나선형·V모델·애자일

1-2. 애자일 기법 — XP, Scrum, Lean

  • XP(eXtreme Programming) 5가지 핵심가치: 의사소통(Communication)·단순성(Simplicity)·용기(Courage)·존중(Respect)·피드백(Feedback). 기법: Pair Programming, TDD, 리팩토링, 사용자 스토리, 지속적 통합(CI).
  • Scrum — Sprint(2~4주), 제품책임자(PO)·스크럼마스터(SM)·개발팀. 일일 스크럼, 백로그.
  • Lean — 7원칙: 낭비 제거, 품질 내재화, 지식 창출, 늦은 결정, 빠른 인도, 사람 존중, 전체 최적화.
XP는 "테스트를 수행하지 않는다" → 절대 틀린 말. XP는 TDD(테스트 주도 개발) 기반!

1-3. 요구사항 분석

  • 기능 요구 — 시스템이 수행해야 할 기능 (예: 조회·입금·송금).
  • 비기능 요구 — 성능, 품질, 보안, 안전성, 응답시간 등. 시험 단골: "화면이 3초 내 보여야 한다" = 비기능.
  • 분석 도구: 자료흐름도(DFD), 자료사전(DD), 소단위명세서(Mini-Spec).

1-5. HIPO · N-S Chart

  • HIPO — 하향식 SW 개발 문서화 도구. 가시적 도표(Visual Table of Contents) / 총체적 도표(Overview) / 세부적 도표(Detail). (함정: "상향식" 이라고 하면 틀림)
  • N-S Chart(Nassi-Schneiderman) — 논리 중심의 도형식 표현. 화살표 사용 X, 박스 구조(연속·선택·반복)로 표현. "주로 화살표 사용" = 틀림.

1-6. UML (Unified Modeling Language) HOT

OMG 제정. 14종 다이어그램. 구조(정적) 7 / 행위(동적) 7.

UML 4대 대표 다이어그램 — Use Case / Class / Sequence / Activity
구조(정적)행위(동적)
Class, Object, Component, Deployment, Package, Composite Structure, Profile Use case, Sequence, Activity, State, Communication(Collaboration), Interaction Overview, Timing
  • Class Diagram — 클래스·속성·연산·관계(연관/일반화/의존/실체화).
  • Use Case Diagram — 사용자(Actor) 관점 기능. 관계: 포함(include)·확장(extend)·일반화.
  • Sequence Diagram — 객체 간 메시지 순서(시간 흐름). 구성: 생명선(Lifeline)·실행(Activation)·메시지·객체. (함정: "확장"은 구성요소 아님)
  • State Diagram — 한 객체의 상태 변화.
  • Activity Diagram — 처리 흐름, 워크플로.
기출 함정: "State Diagram이 객체 간 메시지 교환" = 틀림 (그건 Sequence임).

1-7. 럼바우(Rumbaugh) 객체지향 분석 HOT

순서: 객체 → 동적 → 기능 (OMT)
객체 모델링(Object) — 객체 다이어그램, 정보 모델링
동적 모델링(Dynamic) — 상태 다이어그램, 시간 흐름
기능 모델링(Functional) — DFD, 자료 흐름
"객동기" = 객체→동적→기능 (축구감독 이름처럼 외우기)

1-8. 객체지향 · 캡슐화·상속·다형성

  • 캡슐화(Encapsulation) — 데이터 + 연산 묶음. 정보 은닉의 핵심. 변경 시 영향 최소화.
  • 상속(Inheritance) — 상위 속성/연산 물려받기.
  • 다형성(Polymorphism) — 같은 메시지에 다른 반응(오버로딩/오버라이딩).
  • 추상화(Abstraction) — 복잡도 감소, 공통 특성만 표현.
  • 메시지(Message) — 객체에 행위를 지시하는 명령.

1-9. SOLID 객체지향 설계 원칙 HOT

  • SRP 단일책임 — 클래스는 하나의 책임만.
  • OCP 개방폐쇄 — 확장엔 열리고 수정엔 닫힘.
  • LSP 리스코프치환 — 서브타입은 상위타입 자리에 교체 가능.
  • ISP 인터페이스분리 — 사용하지 않는 인터페이스 의존 금지. "클라이언트가 사용하지 않는 메소드와 의존관계 X".
  • DIP 의존역전 — 추상에 의존, 구체에 의존 X. "추상성 낮은 클래스 → 높은 클래스에 의존".
SOLID = S·O·L·I·D 순서대로

1-10. 결합도 & 응집도 HOT

결합도 · 응집도 스펙트럼 - 좋은 모듈 = 왼쪽으로
결합도(Coupling): 약할수록 좋음
자료(Data) < 스탬프(Stamp) < 제어(Control) < 외부(External) < 공통(Common) < 내용(Content)
응집도(Cohesion): 강할수록 좋음
기능(Functional) > 순차(Sequential) > 교환/통신(Communicational) > 절차(Procedural) > 시간(Temporal) > 논리(Logical) > 우연(Coincidental)
결합도 약→강: "자스제외공내" · 응집도 강→약: "기순교절시논우"
  • 좋은 모듈 = 결합도 낮고 응집도 높음.
  • Cohesion(응집도), Coupling(결합도) 영단어도 바로 구분 가능하게.

1-15. 소프트웨어 아키텍처 HOT

아키텍처 4+1 뷰

  • 논리 뷰(Logical) — 기능적 요구사항. 클래스·객체.
  • 프로세스 뷰(Process) — 비기능 요구, 성능·확장성·동시성.
  • 구현(Implementation) 뷰 — 개발 환경, 컴포넌트·라이브러리.
  • 배치(Deployment) 뷰 — 물리적 노드·시스템 배치.
  • 유스케이스(Use Case) 뷰 — 중심. 시나리오.

대표 아키텍처 패턴

  • 계층(Layers) — OSI, 3-tier(Presentation·Business·Data). 상위 계층이 하위 이용.
  • Client-Server — 요청/응답. 중앙집중.
  • MVC(Model-View-Controller) — UI 분리. Model 데이터/로직, View 표시, Controller 사용자 입력 중계.
  • Pipe-Filter — 데이터 스트림 단방향 처리(컴파일러·셸 파이프).
  • Broker — 분산 컴포넌트 간 중개.
  • Peer-to-Peer — 동등한 노드 간 통신.
  • Event-driven(이벤트 기반) — 발행/구독.
  • Master-Slave — 결함 허용, 계산 병렬화.
  • Blackboard — 음성 인식·추론.
  • Interpreter — 규칙·도메인 특화 언어.

1-18. 소프트웨어 설계 모듈화 지표

  • Fan-In — 모듈을 호출하는 상위 모듈의 수. 많으면 재사용성↑, 설계 시 신중.
  • Fan-Out — 모듈이 호출하는 하위 모듈의 수. 많으면 복잡.
  • 적정 수준: Fan-Out은 7±2 이하.

2-6. McCabe Cyclomatic Complexity

공식: V(G) = E - N + 2 (E=간선, N=노드)
혹은 V(G) = P + 1 (P=결정점 수)

기출 전형: 제어흐름 그래프 주고 V(G) 계산.

2-16. 자료구조 상세 — 스택/큐/데크

스택(Stack) 응용

  • 함수 호출 시 복귀 주소 저장(콜 스택).
  • 수식의 중위→후위/전위 변환.
  • DFS(깊이 우선 탐색).
  • 괄호 검사, Undo/Redo.
  • 역순 문자열 출력.
push(x)  // Top에 삽입
pop()    // Top 제거+반환
peek()   // Top 확인만
isEmpty() / isFull()

TOP 포인터: 공백 상태 TOP=-1, 만원 상태 TOP=n-1. 오버플로·언더플로 주의.

큐(Queue) 응용

  • CPU 스케줄링(FIFO).
  • 인쇄 대기, 버퍼.
  • BFS(너비 우선 탐색).
  • 원형 큐(Circular Queue): 배열 끝에서 처음으로 연결. 공간 낭비 방지.
  • 우선순위 큐(Priority Queue): 우선순위 높은 요소 먼저 제거. 힙으로 구현.
  • 데크(Deque): 양쪽 모두 삽입/삭제 가능(Double-ended Queue).

2-17. 연결 리스트(Linked List)

  • 단순 연결 리스트 — 한 방향.
  • 이중 연결 리스트 — 양방향 탐색, 메모리↑.
  • 환형 연결 리스트 — 마지막이 처음을 가리킴.

장점: 삽입/삭제 용이(O(1)), 배열 대비 크기 제한 없음.
단점: 임의접근 불가(순차 탐색 O(n)), 포인터 공간 필요.

2-18. 그래프(Graph)

  • 표현: 인접 행렬 O(V²) / 인접 리스트 O(V+E).
  • 탐색: DFS(스택), BFS(큐).
  • 최단경로: Dijkstra(음수 간선 X), Bellman-Ford, Floyd-Warshall.
  • 최소 신장 트리(MST): Kruskal(간선 정렬), Prim(정점 확장).
  • 위상 정렬(Topological Sort) — DAG에서 선후관계.

2-19. 트리 — 이진 검색 트리·AVL·힙

  • 이진 검색 트리(BST) — 좌 < 루트 < 우. 검색 O(log n)~O(n).
  • AVL 트리 — 균형 이진 트리. 좌우 높이 차 ≤ 1.
  • B-Tree·B+Tree — 다진 균형 트리. DB·파일 시스템 인덱스.
  • 힙(Heap) — 완전 이진 트리. 최대 힙(Max Heap) 부모 ≥ 자식, 최소 힙(Min Heap) 부모 ≤ 자식.

이진트리 최대 노드 수: 깊이 k인 포화 이진트리 = 2^k − 1. 완전 이진트리 자식 인덱스: 부모 i의 좌=2i, 우=2i+1 (1-based).

2-20. 알고리즘 설계 기법

  • 분할정복(Divide and Conquer) — 문제 분할 → 해결 → 병합. 병합정렬·퀵정렬.
  • 동적계획법(DP) — 중복 부분 문제 + 최적 부분 구조. 피보나치·배낭문제.
  • 탐욕(Greedy) — 매 순간 최적 선택. 거스름돈·크루스칼·다익스트라.
  • 백트래킹(Backtracking) — DFS + 가지치기. N-Queen·미로.
  • 브루트포스(Brute Force) — 모든 경우 탐색.
  • 분기한정(Branch and Bound) — 가능성 있는 분기만 탐색.

2-22. 테스트 커버리지 HOT

  • 문장 커버리지(Statement) — 모든 문장 최소 1회.
  • 분기/결정 커버리지(Decision/Branch) — 모든 분기 true·false 1회.
  • 조건 커버리지(Condition) — 각 조건이 true·false 1회.
  • 조건/결정 커버리지 — 조건+결정 모두.
  • 변형 조건/결정(MC/DC) — 각 조건이 독립적으로 결정에 영향.
  • 다중 조건 커버리지 — 모든 조건 조합.
  • 경로 커버리지 — 모든 가능 경로.

강도: 문장 < 분기 < 조건 < 조건/결정 < MC/DC < 다중조건 < 경로.

2-23. 테스트 오라클

  • 참(True) 오라클 — 모든 입력에 대한 기대값 존재.
  • 샘플링(Sampling) 오라클 — 일부 입력만 기대값 제공.
  • 휴리스틱(Heuristic) 오라클 — 샘플링 + 나머지는 경험적 추정.
  • 일관성 검사(Consistent) 오라클 — 변경 전/후 값이 동일한지.

2-24. 테스트 기법 — 블랙박스 심화

  • 동등 분할(Equivalence Partitioning) — 유효/무효 클래스 나누고 대표값 테스트.
  • 경계값 분석(BVA · Boundary Value Analysis) — 경계·경계±1. 대부분 버그는 경계에서 발생.
  • 원인-결과 그래프 — 입력과 결과의 관계를 그래프로.
  • 오류 예측(Error Guessing) — 경험 기반 예측.
  • 비교(Comparison) — 여러 버전 비교.
  • 결정 테이블(Decision Table) — 조건 조합 정리.

2-25. 화이트박스 테스트 심화

  • 제어 흐름 검사 — CFG 기반.
  • 데이터 흐름 검사 — 정의·사용(DU) 쌍.
  • 루프 검사 — 단순·중첩·연결 루프.
  • 조건 검사.

2-29. 해싱 충돌 해결 심화

  • 개방 주소법(Open Addressing) — 다음 빈 곳 찾음.
    • 선형 조사(Linear) — +1,+2,...
    • 이차 조사(Quadratic) — +1²,+2²,...
    • 이중 해싱(Double) — 2차 해시 함수.
  • 체이닝(Chaining) — 같은 해시값은 연결리스트로.
  • 합병 해싱(Coalesced) — 개방+체이닝 혼합.

3-1. 데이터베이스 설계 단계 HOT

단계수행 작업

3-2. E-R 모델

  • 개체(Entity) = 사각형, 속성(Attribute) = 타원, 관계(Relationship) = 마름모, 연결 = 선.
  • 개체·속성·관계 타입을 기본 요소로 현실세계를 표현.

3-3. 관계형 데이터베이스 용어 HOT

  • 릴레이션(Relation) = 테이블.
  • 튜플(Tuple) = 행(Row) = 레코드. 카디널리티(Cardinality) = 튜플 수.
  • 속성(Attribute) = 열(Column) = 필드. 차수(Degree) = 속성 수.
  • 도메인(Domain) = 속성이 취할 수 있는 값의 집합.

3-4. 키(Key) 종류

  • 후보키(Candidate) — 유일성 + 최소성 만족.
  • 기본키(Primary) — 후보키 중 하나 선택, NULL 불가.
  • 대체키(Alternate) — 후보키 중 기본키 제외.
  • 슈퍼키(Super) — 유일성만 만족(최소성 X).
  • 외래키(Foreign) — 다른 릴레이션의 기본키 참조.
참조 무결성: 외래키 값은 참조 테이블의 기본키에 존재하거나 NULL.

3-5. 무결성 제약 HOT

  • 개체 무결성 — 기본키 NULL 불가, 중복 불가.
  • 참조 무결성 — 외래키 ↔ 참조 릴레이션 기본키 대응.
  • 도메인 무결성 — 속성 값이 도메인에 속함.
  • 사용자 정의 무결성 — 업무규칙.
  • NULL 무결성 / Key(유일성) 무결성.
"보안 무결성·정보 무결성" 같은 용어는 기출에서 가짜 선지!

3-6. 정규화(Normalization) HOT

정규형조건
1NF모든 속성이 원자값 (도메인 분할 불가)
2NF1NF + 기본키에 완전 함수 종속 (부분 종속 제거)
3NF2NF + 이행 종속 제거
BCNF3NF + 모든 결정자가 후보키
4NF다치 종속 제거
5NF조인 종속 제거
"도부이결다조" = 도메인원자·부분종속제거·이행종속제거·결정자후보키·다치·조인

역정규화(Denormalization) — 성능을 위해 일부러 중복 허용.

3-7. 관계 대수 & 관계 해석

  • 관계 대수: 절차적 언어. 원하는 결과를 얻기 위한 연산 명시.
  • 관계 해석: 비절차적, 서술적.

순수 관계 연산: SELECT(σ) · PROJECT(π) · JOIN(⋈) · DIVISION(÷)

일반 집합 연산: 합집합(∪)·교집합(∩)·차집합(−)·곱집합(×)

3-8. SQL — DDL/DML/DCL HOT

분류명령어
DDL(정의)CREATE · ALTER · DROP · TRUNCATE · RENAME
DML(조작)SELECT · INSERT · UPDATE · DELETE
DCL(제어)GRANT · REVOKE · COMMIT · ROLLBACK · SAVEPOINT
기출: "DML에 해당? INSERT·GRANT" → GRANT는 DCL!

3-9. SQL 문법 핵심

-- 기본
SELECT 컬럼 FROM 테이블 WHERE 조건
GROUP BY 그룹 HAVING 집계조건
ORDER BY 정렬 [ASC|DESC];

-- UPDATE (SET 위치 주의)
UPDATE 사원 SET 사원코드=사원코드+1000

-- LIKE 패턴

WHERE 점수 IS NOT NULL

-- JOIN

SELECT * FROM A LEFT OUTER JOIN B ON A.id = B.id;

COUNT(*), SUM(col), AVG(col), MAX(col), MIN(col)

3-10. VIEW(뷰) 특징

  • CREATE VIEW로 정의, DROP으로 제거.
  • 논리적 독립성 제공, 물리적으로 존재하지 않음(가상 테이블).
  • 보안 측면에서 DBA가 활용 가능(특정 컬럼만 노출).
  • ALTER 불가(수정은 DROP 후 재생성).
함정: "뷰는 저장장치 내에 물리적으로 존재한다" = 틀림.

3-11. 트랜잭션 ACID HOT

  • Atomicity 원자성 — 모두 반영 or 모두 취소.
  • Consistency 일관성 — 실행 전후 DB 일관된 상태.
  • Isolation 고립성 — 실행 중 중간 결과 다른 트랜잭션 접근 X.
  • Durability 지속성 — 완료 후 영구 반영.

트랜잭션 상태: 활동(Active)→부분완료→완료(Committed)/실패→철회(Aborted, Rollback).

3-12. 병행제어(Concurrency) · 로킹

  • 로킹 단위가 작으면: 병행성↑, 오버헤드↑.
  • 단위가 크면: 공유도↓, 오버헤드↓.
  • 2PL(Two-Phase Locking): 확장(Lock만) → 축소(Unlock만) 단계.
  • 타임스탬프 순서 기법, 낙관적(Optimistic), 다중 버전 기법(MVCC).
함정: "로킹 단위 커지면 공유도 증가" = 틀림 (공유도 감소).

3-13. 트리거(Trigger)

  • INSERT/UPDATE/DELETE 이벤트에 자동 실행되는 절차형 SQL.
  • 무결성 유지, 로그 기록 용도.
  • CREATE TRIGGER 로 생성.

3-14. 파티셔닝(Partitioning)

  • 하나의 테이블을 작은 논리적 단위로 분할.
  • 가용성·성능 향상.
  • 종류: Range·List·Hash·Composite.

3-15. 인덱스(Index)

  • B-Tree, B+Tree, Hash, Bitmap 인덱스.
  • 검색 속도 향상, 갱신 시 오버헤드.
  • 클러스터드(데이터 정렬 저장) vs 논클러스터드.

3-16. 회복(Recovery) 기법

  • 로그 기반: 지연 갱신, 즉시 갱신.
  • 검사점(Checkpoint) 기법.
  • 그림자 페이징(Shadow Paging).
  • REDO: 커밋된 트랜잭션 재실행. UNDO: 미완료 트랜잭션 취소.

3-17. 분산 데이터베이스 목표

투명성: 위치·중복·병행·장애 투명성.

  • 위치 투명성 — 데이터 위치 몰라도 접근 가능.
  • 중복 투명성 — 중복 데이터 존재 여부 무관.
  • 병행 투명성 — 다중 트랜잭션 동시성 보장.
  • 장애 투명성 — 일부 고장시 일관성 유지.
  • 분할 투명성 — 테이블 분할 숨김.

3-18. SQL 심화 — GROUP BY · HAVING · WITH

-- 부서별 평균 급여가 300 이상인 부서
SELECT 부서, AVG(급여) AS 평균
FROM 사원
GROUP BY 부서
HAVING AVG(급여) >= 300;

-- 서브쿼리
SELECT 이름 FROM 사원
WHERE 급여 > (SELECT AVG(급여) FROM 사원);

-- EXISTS
SELECT * FROM 부서 D
WHERE EXISTS (SELECT 1 FROM 사원 S WHERE S.부서=D.부서);

-- IN
SELECT * FROM 사원
WHERE 부서 IN ('인사','개발');

-- CASE
SELECT 이름,
  CASE WHEN 급여 >= 500 THEN 'A'
       WHEN 급여 >= 300 THEN 'B'
       ELSE 'C' END AS 등급
FROM 사원;

3-19. JOIN 종류 상세 HOT

  • INNER JOIN — 양쪽 매칭.
  • LEFT OUTER JOIN — 왼쪽 모두 + 오른쪽 매칭.
  • RIGHT OUTER JOIN — 오른쪽 모두 + 왼쪽 매칭.
  • FULL OUTER JOIN — 양쪽 모두.
  • CROSS JOIN — 카티션 곱.
  • SELF JOIN — 자기 자신과 조인.
  • NATURAL JOIN — 동일 이름 컬럼 자동 매칭.
SELECT A.이름, B.부서명
FROM 사원 A INNER JOIN 부서 B ON A.부서ID = B.부서ID;

SELECT A.이름, B.부서명
FROM 사원 A LEFT JOIN 부서 B ON A.부서ID = B.부서ID;

3-20. 관계형 연산 — SELECT·PROJECT·JOIN·DIVISION

  • σ (Select, 선택) — 조건 만족 튜플. σ급여≥500(사원).
  • π (Project, 추출) — 특정 속성만. π이름,급여(사원).
  • ⋈ (Join) — 두 릴레이션 결합.
  • ÷ (Division) — R ÷ S : S의 모든 튜플과 연관된 R의 튜플.
  • ∪ 합집합·∩ 교집합·− 차집합·× 카티션 곱.

3-21. 함수 종속 · 이상(Anomaly)

  • 함수 종속(FD) X → Y: X가 Y를 결정.
  • 완전 함수 종속 — Y가 X 전체에 종속.
  • 부분 함수 종속 — Y가 X의 일부에만 종속.
  • 이행 함수 종속 — X→Y, Y→Z이면 X→Z.
  • 삽입 이상 — 불필요 값 없인 삽입 불가.
  • 삭제 이상 — 한 행 삭제 시 다른 정보도 삭제.
  • 갱신 이상 — 일부만 갱신되어 불일치.

3-22. 뷰(View) · 인덱스 · 시퀀스

  • 뷰의 제거: DROP VIEW 이름 [CASCADE|RESTRICT];
  • 인덱스 생성: CREATE [UNIQUE] INDEX idx_name ON 테이블(컬럼);
  • 시퀀스: CREATE SEQUENCE seq_name START WITH 1 INCREMENT BY 1;

3-23. SQL DCL 상세

GRANT SELECT, INSERT ON 사원 TO user1 WITH GRANT OPTION;

COMMIT;    -- 반영 확정

SAVEPOINT s1;

3-24. DB 병행 제어 기법 심화

  • 로킹(Locking) — 공유락(S) / 배타락(X). 2PL로 직렬성 보장.
  • 타임스탬프 순서 — 트랜잭션에 시간 부여, 순서 위반시 철회.
  • 낙관적 병행제어 — 충돌 적을 때 유리.
  • 다중버전(MVCC) — 읽기는 이전 버전, 쓰기는 새 버전.

문제: 갱신 분실, 모순성, 연쇄 철회, 비완료 의존성. → 로킹으로 해결.

3-25. 회복 기법 상세

  • 지연 갱신(Deferred Update) — 커밋까지 DB 반영 X. UNDO 불필요, REDO만.
  • 즉시 갱신(Immediate Update) — 바로 반영. UNDO + REDO 필요.
  • 검사점 회복 — 검사점 이후만 회복.
  • 그림자 페이징 — 복제본 관리.

3-26. 데이터 웨어하우스 · 마트 · OLAP

  • DW(Data Warehouse) — 주제 지향·통합·시계열·비휘발적.
  • Data Mart — 특정 부서용 소규모 DW.
  • ETL — Extract·Transform·Load.
  • OLAP — 다차원 분석(MOLAP/ROLAP/HOLAP). 연산: Roll-up·Drill-down·Slicing·Dicing·Pivoting.
  • OLTP — 온라인 트랜잭션 처리.

3-27. 데이터 모델링 단계별 상세

개념적 설계

E-R 다이어그램. DBMS와 독립. 사용자 관점 모델링. 현실→추상화.

논리적 설계

E-R → 관계 스키마 변환. 정규화 수행. 특정 DBMS에 종속적 설계.

물리적 설계

저장 레코드 형식, 접근 경로, 인덱스, 블록 크기 결정. 성능 목표 달성.

3-28. 스키마 3계층

독립성: 외부↔개념 = 논리적 독립성 · 개념↔내부 = 물리적 독립성
  • 외부 스키마(서브스키마) — 사용자·응용별 뷰.
  • 개념 스키마 — 전체 논리적 구조. DB 관리자 관점.
  • 내부 스키마 — 저장 구조·인덱스. 물리 관점.

데이터 독립성: 논리적(외부↔개념)·물리적(개념↔내부).

"관계형 데이터 모델에서 한 릴레이션의 ( )는 참조되는 릴레이션의 기본 키와 대응된다"의 ( )는?

외래키(Foreign Key).

1NF에서 2NF가 되기 위한 조건은?

1NF를 만족하고 키가 아닌 모든 속성이 기본키에 완전 함수적 종속.

병행제어 로킹 설명 중 옳지 않은 것은? (4) 로킹의 단위가 커지면 데이터베이스 공유도가 증가한다.

④ 틀림. 단위가 커지면 공유도 감소, 병행성 감소, 오버헤드 감소.

UPDATE 구문을 바르게 작성한 것은? (부서 '인사' AND 사원코드≤3000 의 사원코드를 1000 증가)

UPDATE 사원 SET 사원코드=사원코드+1000 WHERE 부서='인사' AND 사원코드<=3000;

DML에 해당하는 것으로만 나열된 것은? ㉠CREATE ㉡DROP ㉢INSERT ㉣GRANT

㉢ INSERT만 DML. CREATE·DROP=DDL, GRANT=DCL.

뷰에 대한 설명으로 옳지 않은 것은? (4) 뷰는 저장장치 내에 물리적으로 존재한다.

④ 틀림. 뷰는 가상 테이블이며 물리적 저장 X.

트랜잭션의 특성 중 "부분 실행시 전혀 반영되지 않음"을 의미하는 것은?

원자성(Atomicity). All or Nothing.

분산 DB에서 사용자가 데이터의 실제 위치를 알 필요가 없는 특성은?

위치 투명성.
